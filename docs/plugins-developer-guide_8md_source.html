1 # 插件的工作原理
2 
3 插件是一种在不需要改动并重新编译主程序本身的情况下去扩展主程序功能的一种机制。\
4 dde-dock 插件是根据 Qt 插件标准所开发的共享库文件(`so`)，通过实现 Qt 的插件标准和 dde-dock 提供的接口，共同完成 dde-dock 的功能扩展。\
5 可以通过以下链接查看关于 Qt 插件更详细的介绍：
6 
7 [https://wiki.qt.io/Plugins](https://wiki.qt.io/Plugins)\
8 [https://doc.qt.io/qt-5/plugins-howto.html](https://doc.qt.io/qt-5/plugins-howto.html)
9 
10 ## dde-dock 插件加载流程
11 
12 在 dde-dock 启动时会跑一个线程去检测目录`/usr/lib/dde-dock/plugins`下的所有文件，并检测是否是一个正常的动态库文件，如果是则尝试加载。尝试加载即检测库文件的元数据，插件的元数据定义在一个 JSON 文件中，这个后文会介绍，如果元数据检测通过就开始检查插件是否实现了 dde-dock 指定的接口，这一步也通过之后就会开始初始化插件，获取插件提供的控件，进而将控件显示在任务栏上。
13 
14 ## 接口列表
15 
16 这里先列出 dde-dock 都提供了哪些接口，可作为一个手册查看，注意，为 dde-dock 编写插件并不是要实现所有接口，这些接口提供了 dde-dock 允许各种可能的功能，插件开发者可以根据自己的需求去实现自己需要的接口。后续的插件示例也将会用到这里列出的部分接口。
17 
18 接口定义的文件一般在系统的如下位置：
19 ```
20 /usr/include/dde-dock/pluginproxyinterface.h
21 /usr/include/dde-dock/pluginsiteminterface.h
22 ```
23 
24 ### PluginItemInterface
25 
26 ***只有标明`必须实现`的接口是必须要由插件开发者实现的接口，其他接口如果不需要对应功能可不实现。***
27 
28 PluginsItemInterface 中定义的接口除了displayMode 和 position（历史遗留），从插件的角度来看都是被动的，只能等待被任务栏的插件机制调用。
29 
30 |名称|简介|
31 |-|-|
32 |pluginName | 返回插件名称，用于在 dde-dock 内部管理插件时使用 `必须实现`|
33 |pluginDisplayName | 返回插件名称，用于在界面上显示|
34 |init | 插件初始化入口函数，参数 proxyInter 可认为是主程序的进程 `必须实现`|
35 |itemWidget | 返回插件主控件，用于显示在 dde-dock 面板上 `必须实现`|
36 |itemTipsWidget | 返回鼠标悬浮在插件主控件上时显示的提示框控件|
37 |itemPopupApplet | 返回鼠标左键点击插件主控件后弹出的控件|
38 |itemCommand | 返回鼠标左键点击插件主控件后要执行的命令数据|
39 |itemContextMenu | 返回鼠标右键点击插件主控件后要显示的菜单数据|
40 |invokedMenuItem | 菜单项被点击后的回调函数|
41 |itemSortKey | 返回插件主控件的排序位置|
42 |setSortKey | 重新设置主控件新的排序位置（用户拖动了插件控件后）|
43 |itemAllowContainer | 返回插件控件是否允许被收纳|
44 |itemIsInContainer | 返回插件是否处于收纳模式（仅在 itemAllowContainer 为 true 时有作用）|
45 |setItemIsInContainer | 更新插件是否处于收纳模式的状态（仅在 itemAllowContainer 主 true 时有作用）|
46 |pluginIsAllowDisable | 返回插件是否允许被禁用（默认不允许被禁用）|
47 |pluginIsDisable | 返回插件当前是否处于被禁用状态|
48 |pluginStateSwitched | 当插件的禁用状态被用户改变时此接口被调用|
49 |displayModeChanged | dde-dock 显示模式发生改变时此接口被调用|
50 |positionChanged | dde-dock 位置变化时时此接口被调用|
51 |refreshIcon | 当插件控件的图标需要更新时此接口被调用|
52 |displayMode | 用于插件主动获取 dde-dock 当前的显示模式|
53 |position | 用于插件主动获取 dde-dock 当前的位置|
54 
55 ### PluginProxyInterface
56 
57 由于上面的接口对于插件来说都是被动的，即插件本身无法确定这些接口什么时刻会被调用，很明显这对于插件机制来说是不完整的，因此便有了 PluginProxyInterface，它定义了一些让插件主动调用以控制 dde-dock 的一些行为的接口。PluginProxyInterface 的具体实例可以认为是抽象了的 dde-dock 主程序，或者是 dde-dock 中所有插件的管理员，这个实例将会通过 PluginItemInterface 中的 `init` 接口传递给插件，因此在上述 `init` 接口中总是会先把这个传入的对象保存起来以供后续使用。
58 
59 |名称|简介|
60 |-|-|
61 |itemAdded | 向 dde-dock 添加新的主控件（一个插件可以添加多个主控件它们之间使用`ItemKey`区分）|
62 |itemUpdate | 通知 dde-dock 有主控件需要更新|
63 |itemRemoved | 从 dde-dock 移除主控件|
64 |requestWindowAutoHide | 设置 dde-dock 是否允许隐藏，通常被用在任务栏被设置为智能隐藏或始终隐藏而插件又需要让 dde-dock 保持显示状态来显示一些重要信息的场景下|
65 |requestRefreshWindowVisible | 通知 dde-dock 更新隐藏状态|
66 |requestSetAppletVisible | 通知 dde-dock 显示或隐藏插件的弹出面板（鼠标左键点击后弹出的控件）|
67 |saveValue | 统一的配置保存函数|
68 |getValue | 统一的配置读取函数|

107 ### cmake 配置文件
108 
109 `CMakeLists.txt` 是 cmake 命令要读取的配置文件，用于管理整个项目的源文件，依赖，构建等等，其内容如下：
110 
111 > 以`#`开头的行是注释，用于介绍相关命令，对创建一份新的 CMakeLists.txt 文件会有所帮助，目前可以简单地过一遍
112 
113 ``` cmake
114 # 学习 cmake 时建议直接从命令列表作为入口，遇到不清楚意思的命令都可以在此处查阅：
115 # https://cmake.org/cmake/help/latest/manual/cmake-commands.7.html
116 # 另外下面时完整的文档入口：
117 # https://cmake.org/cmake/help/latest/
118 
119 # 设置运行被配置所需的 cmake 最低版本
120 cmake_minimum_required(VERSION 3.11)
121 
122 # 使用 set 命令设置一个变量
123 set(PLUGIN_NAME "home_monitor")
124 
125 # 设置项目名称
126 project(${PLUGIN_NAME})
127 
128 # 启用 qt moc 的支持
129 set(CMAKE_AUTOMOC ON)
130 # 启用 qrc 资源文件的支持
131 set(CMAKE_AUTORCC ON)
132 
133 # 指定所有源码文件
134 # 使用了 cmake 的 file 命令，递归查找项目目录下所有头文件和源码文件，
135 # 并将结果放入 SRCS 变量中，SRCS 变量可用于后续使用
136 file(GLOB_RECURSE SRCS "*.h" "*.cpp")
137 
138 # 指定要用到的库
139 # 使用了 cmake 的 find_package 命令，查找库 Qt5Widgets 等，
140 # REQUIRED 参数表示如果没有找到则报错
141 # find_package 命令在找到并加载指定的库之后会设置一些变量，
142 # 常用的有：
143 # <库名>_FOUND 是否找到（Qt5Widgets_FOUND）
144 # <库名>_DIR 在哪个目录下找到的（Qt5Widgets_DIR）
145 # <库名>_INCLUDE_DIRS 有哪些头文件目录（Qt5Widgets_INCLUDE_DIRS）
146 # <库名>_LIBRARIES 有哪些库文件（Qt5Widgets_LIBRARIES）
147 find_package(Qt5Widgets REQUIRED)
148 find_package(DtkWidget REQUIRED)
149 
150 # find_package 命令还可以用来加载 cmake 的功能模块
151 # 并不是所有的库都直接支持 cmake 查找的，但大部分都支持了 pkg-config 这个标准，
152 # 因此 cmake 提供了间接加载库的模块：FindPkgConfig， 下面这行命令表示加载 FindPkgConfig 模块，
153 # 这个 cmake 模块提供了额外的基于 “pkg-config” 加载库的能力
154 # 执行下面的命令后后会设置如下变量，不过一般用不到：
155 # PKG_CONFIG_FOUND pkg-config 可执行文件是否找到了
156 # PKG_CONFIG_EXECUTABLE pkg-config 可执行文件的路径
157 # PKG_CONFIG_VERSION_STRING pkg-config 的版本信息
158 find_package(PkgConfig REQUIRED)
159 
160 # 加载 FindPkgConfig 模块后就可以使用 pkg_check_modules 命令加载需要的库
161 # pkg_check_modules 命令是由 FindPkgConfig 模块提供的，因此要使用这个命令必须先加载 FindPkgConfig 模块。
162 # 执行 pkg_check_modules 命令加载库也会设置一些类似执行 find_package 加载库后设置的变量：
163 # DdeDockInterface_FOUND
164 # DdeDockInterface_INCLUDE_DIRS
165 # DdeDockInterface_LIBRARIES
166 # 还有有另外的一些变量以及更灵活的用法，比如一次性查找多个库，这些请自行查找 cmake 文档学习。
167 pkg_check_modules(DdeDockInterface REQUIRED dde-dock)
168 
169 # add_definitions 命令用于声明/定义一些编译/预处理参数
170 # 根据 cmake 文档描述此命令已经有另外几个功能划分的更为细致的命令所取代，具体请查阅文档
171 # 在我们这里的例子应该使用较新的 add_compile_definitions 命令，不过为了保持与 dock 已有插件一致，
172 # 暂时仍然使用 add_definitions，add_definitions 的语法很简单就是直接写要定义的 flag 并在前面加上 "-D" 即可
173 # 括号中的 ${QT_DEFINITIONS} 变量会在执行 cmake 时展开为它的值，这个变量属于历史遗留，应该是在 qt3/qt4 时有用，
174 # 基于 qt5 或更高版本的新插件不必使用此变量。要查看 qt5 的库定义了哪些变量应该查看变量：${Qt5Widgets_DEFINITIONS}
175 add_definitions("${QT_DEFINITIONS} -DQT_PLUGIN")
176 
177 # 新增一个编译目标
178 # 这里使用命令 add_library 来表示本项目要生成一个库文件目标，
179 # 类似的还有命令 add_executable 添加一个可执行二进制目标，甚至 add_custom_target(使用较少) 添加自定义目标
180 # SHARED 表示生成的库应该是动态库，
181 # 变量 ${PLUGIN_NAME} 和 ${SRCS} 都是前面处理好的，
182 # 另外 qrc 资源文件也应该追加在后面以编译进目标中。
183 add_library(${PLUGIN_NAME} SHARED ${SRCS} home_monitor.qrc)
184 
185 # 设置目标的生成位置，这里表示生成在执行 make 的目录,
186 # 另外还有很多可用于设置的属性，可查阅 cmake 文档。
187 set_target_properties(${PLUGIN_NAME} PROPERTIES LIBRARY_OUTPUT_DIRECTORY ./)
188 
189 # 设置目标要使用的 include 目录，即头文件目录
190 # 变量 ${DtkWidget_INCLUDE_DIRS} 是在前面执行 find_package 命令时引入的
191 # 当出现编译失败提示找不到某些库的头文件时应该检查此处是否将所有需要的头文件都包含了
192 target_include_directories(${PLUGIN_NAME} PUBLIC
193  ${Qt5Widgets_INCLUDE_DIRS}
194  ${DtkWidget_INCLUDE_DIRS}
195  ${DdeDockInterface_INCLUDE_DIRS}
196 )
197 
198 # 设置目标要使用的链接库
199 # 变量 ${DtkWidget_LIBRARIES} 和 ${Qt5Widgets_LIBRARIES} 是在前面执行 find_package 命令时引入的
200 # 当出现运行时错误提示某些符号没有定义时应该检查此处是否将所有用的库都写在了这里
201 target_link_libraries(${PLUGIN_NAME} PRIVATE
202  ${Qt5Widgets_LIBRARIES}
203  ${DtkWidget_LIBRARIES}
204  ${DdeDockInterface_LIBRARIES}
205 )
206 
207 # 设置安装路径的前缀(默认为"/usr/local")
208 set(CMAKE_INSTALL_PREFIX "/usr")
209 
210 # 设置执行 make install 时哪个目标应该被 install 到哪个位置
211 install(TARGETS ${PLUGIN_NAME} LIBRARY DESTINATION lib/dde-dock/plugins)
212 ```

214 ### 元数据文件
215 
216 `home_monitor.json`文件是插件的元数据文件，指明了当前插件所使用的 dde-dock 的接口版本，dde-dock 在加载此插件时，会检测自己的接口版本是否与插件的接口版本一致，当双方的接口版本不一致或者不兼容时，dde-dock 为了安全将阻止加载对应的插件。另外，元数据文件是在源代码中使用特定的宏加载到插件中的。
217 
218 在 dde-dock 内建的插件代码中，可以找到当前具体的接口版本，目前最新的版本是 `1.2` 。
219 
220 ``` json
221 {
222  "api": "1.2"
223 }
224 ```
225 
226 另外（可选的）还支持指定一个 dbus 服务，dock 在加载插件时会检查此插件所依赖的 dbus 服务，如果服务没有启动则不会初始化这个插件，直到服务启动，如下表示依赖 dbus 地址为 "com.deepin.daemon.Network" 的 dbus 服务。
227 
228 ``` json
229 {
230  "api": "1.2",
231  "depends-daemon-dbus-service": "com.deepin.daemon.Network"
232 }
233 ```

235 ### 插件核心类
236 
237 `homemonitorplugin.h` 声明了类 `HomeMonitorPlugin`，它继承（实现）了前面提到的 `PluginItemInterface`，这代表了它是一个实现了 dde-dock 接口的插件。
238 
239 下面是最小化实现了一个 dock 插件的源码，只实现了必须实现的接口，请注意，下文的代码只是为了简述开发一个插件的主要过程，详细的示例代码应该查看 `home-monitor` 目录下的内容。
240 
241 ``` c++
242 #ifndef HOMEMONITORPLUGIN_H
243 #define HOMEMONITORPLUGIN_H
244 
245 #include <dde-dock/pluginsiteminterface.h>
246 
247 #include <QObject>
248 
249 class HomeMonitorPlugin : public QObject, PluginsItemInterface
250 {
251  Q_OBJECT
252  // 声明实现了的接口
253  Q_INTERFACES(PluginsItemInterface)
254  // 插件元数据
255  Q_PLUGIN_METADATA(IID "com.deepin.dock.PluginsItemInterface" FILE "home_monitor.json")
256 
257 public:
258  explicit HomeMonitorPlugin(QObject *parent = nullptr);
259 
260  // 返回插件的名称，必须是唯一值，不可以和其它插件冲突
261  const QString pluginName() const override;
262 
263  // 插件初始化函数
264  void init(PluginProxyInterface *proxyInter) override;
265 
266  // 返回插件的 widget
267  QWidget *itemWidget(const QString &itemKey) override;
268 };
269 
270 #endif // HOMEMONITORPLUGIN_H
271 ```
272

274

276 #include "homemonitorplugin.h"
277

279  : QObject(parent)
280 {
281 
282 }
283 
284 const QString HomeMonitorPlugin::pluginName() const
285 {

287 }
288

292 }
293

419  , m_infoLabel(new QLabel)

423  , m_storageInfo(new QStorageInfo("/home"))
424 {
425  m_infoLabel->setStyleSheet("QLabel {"
426  "color: white;"
427  "}");

429 
430  QVBoxLayout *centralLayout = new QVBoxLayout;
431  centralLayout->addWidget(m_infoLabel);
432  centralLayout->setSpacing(0);
433  centralLayout->setMargin(0);
434 
435  setLayout(centralLayout);
436 
437  // 连接 Timer 超时的信号到更新数据的槽上
438  connect(m_refreshTimer, &QTimer::timeout, this, &InformationWidget::refreshInfo);
439 
440  // 设置 Timer 超时为 10s，即每 10s 更新一次控件上的数据，并启动这个定时器
441  m_refreshTimer->start(10000);
442 
443  refreshInfo();
444 }
445 
446 void InformationWidget::refreshInfo()
447 {
448  // 获取分区总容量
449  const double total = m_storageInfo->bytesTotal();
450  // 获取可用总容量
451  const double available = m_storageInfo->bytesAvailable();
452  // 得到可用百分比
453  const int percent = qRound(available / total * 100);
454 
455  // 更新内容
456  m_infoLabel->setText(QString("Home:\n%1\%").arg(percent));
457 }

485 ## 添加主控件到 dde-dock 面板上
486 
487 在插件核心类的 `init` 方法中获取到了 `PluginProxyInterface` 对象，调用此对象的 `itemAdded` 接口即可实现向 dde-dock 面板上添加项目。
488 
489 第二个 `QString` 类型的参数代表了本插件所提供的主控件的 id，当一个插件提供多个主控件时，不同主控件之间的 id 要保证唯一。
490 
491 ``` c++
492 void HomeMonitorPlugin::init(PluginProxyInterface *proxyInter)
493 {
494  m_proxyInter = proxyInter;
495 
496  m_pluginWidget = new InformationWidget;
497 
498  m_proxyInter->itemAdded(this, pluginName());
499 }
500 ```
501 
502 在调用 `itemAdded` 之后，dde-dock 会在合适的时机调用插件的`itemWidget`接口以获取需要显示的控件。如果插件提供了多个主控件到 dde-dock 上，那么插件核心类应该在 itemWidget 接口中分析参数 itemKey，并返回与之对应的控件对象，当插件只有一个可显示项目时，itemKey 可以忽略 (但不建议忽略)。
503 
504 ``` c++
505 QWidget *HomeMonitorPlugin::itemWidget(const QString &itemKey)
506 {
507  Q_UNUSED(itemKey);
508 
509  return m_pluginWidget;
510 }
511 ```
512

517 ## 支持禁用插件
518 
519 与插件禁用和启用相关的接口有如下三个：
520 
521 - pluginIsAllowDisable
522 - pluginIsDisable
523 - pluginStateSwitched
524 
525 故而在插件的核心类头文件中增加这三个接口的声明：
526 
527 ``` c++
528 bool pluginIsAllowDisable() override;
529 bool pluginIsDisable() override;
530 void pluginStateSwitched() override;
531 ```
532 
533 同时在插件的核心类实现类中增加这三个接口的定义：
534 
535 ``` c++
536 bool HomeMonitorPlugin::pluginIsAllowDisable()
537 {
538  // 告诉 dde-dock 本插件允许禁用
539  return true;
540 }
541 
542 bool HomeMonitorPlugin::pluginIsDisable()
543 {
544  // 第二个参数 “disabled” 表示存储这个值的键（所有配置都是以键值对的方式存储的）
545  // 第三个参数表示默认值，即默认不禁用
546  return m_proxyInter->getValue(this, "disabled", false).toBool();
547 }
548 
549 void HomeMonitorPlugin::pluginStateSwitched()
550 {
551  // 获取当前禁用状态的反值作为新的状态值
552  const bool disabledNew = !pluginIsDisable();
553  // 存储新的状态值
554  m_proxyInter->saveValue(this, "disabled", disabledNew);
555 
556  // 根据新的禁用状态值处理主控件的加载和卸载
557  if (disabledNew) {
558  m_proxyInter->itemRemoved(this, pluginName());
559  } else {
560  m_proxyInter->itemAdded(this, pluginName());
561  }
562 }
563 ```
564 
565 此时就会引入一个新的问题，插件允许被禁用，那么在 dde-dock 启动时，插件有可能处于禁用状态，那么在初始化插件时就不能直接将主控件添加到 dde-dock 中，而是应该判断当前是否是禁用状态，修改接口 init 的实现：
566 
567 ``` c++
568 void HomeMonitorPlugin::init(PluginProxyInterface *proxyInter)
569 {
570  m_proxyInter = proxyInter;
571 
572  m_pluginWidget = new InformationWidget;
573 
574  // 如果插件没有被禁用则在初始化插件时才添加主控件到面板上
575  if (!pluginIsDisable()) {
576  m_proxyInter->itemAdded(this, pluginName());
577  }
578 }
579 ```
580 
581 重新编译、安装、重启 dde-dock，然后 dde-dock 面板上点击鼠标右键查看“插件”子菜单就会看到空白项，点击它将禁用插件，再次点击则启用插件。
582 
583 不过为什么是空白项呢？是因为有一个接口还没有实现：pluginDisplayName
584 
585 在相应文件中分别添加如下内容，来修复这个问题：
586 
587 ``` c++
588 // homemonitorplugin.h
589 
590 const QString pluginDisplayName() const override;
591 ```
592 
593 ``` c++
594 // homemonitorplugin.cpp
595 
596 const QString HomeMonitorPlugin::pluginDisplayName() const
597 {
598  return QString("Home Monitor");
599 }
600 ```

604 ## 支持 hover tip
605 
606 “hover tip” 就是鼠标移动到插件主控件上并悬浮一小段时间后弹出的一个提示框，可以用于显示一些状态信息等待，当然具体用来显示什么完全由插件开发者自己决定，要实现这个功能需要接口：
607 
608 - itemTipsWidget
609 
610 首先在插件核心类中添加一个文本控件作为 tip 控件：

633  m_proxyInter->itemAdded(this, pluginName());

635 }

666 ## 支持 applet
667 
668 上面的 tips 显示的控件在鼠标移开之后就会消失，如果插件需要长时间显示一个窗体及时鼠标离开也会保持显示状态来做一些提示或功能的话那就需要使用 applet，applet 控件在左键点击后显示，点击控件以外的其他地方后消失。
669 
670 applet 控件其实跟 tip 控件一样都是一个普通的 widget，但是可以在 applet 控件中显示交互性的内容，比如按钮，输入框等等。出于篇幅原因这里的 applet 控件就不添加交互性的功能了，只用来显示一些文字，所以依然使用一个 lable 控件。
671 
672 在插件核心类中添加一个文本控件作为 applet 控件：
673 
674 ``` c++
675 // homemonitorplugin.h
676 private:
677  InformationWidget *m_pluginWidget;
678  QLabel *m_tipsWidget;
679  QLabel *m_appletWidget; // new
680 ```
681 
682 在 init 函数中初始化：
683 
684 ``` c++
685 // homemonitorplugin.cpp
686 
687 void HomeMonitorPlugin::init(PluginProxyInterface *proxyInter)
688 {
689  m_proxyInter = proxyInter;
690 
691  m_pluginWidget = new InformationWidget;
692  m_tipsWidget = new QLabel;
693  m_appletWidget = new QLabel; // new
694 
695  // 如果插件没有被禁用则在初始化插件时才添加主控件到面板上
696  if (!pluginIsDisable()) {
697  m_proxyInter->itemAdded(this, pluginName());
698  }
699 }
700 ```

709 
710 ``` c++

713 {
714  Q_UNUSED(itemKey);
715 
716  m_appletWidget->setText(QString("Total: %1G\nAvailable: %2G\nDevice: %3\nVolume: %4\nLabel: %5\nFormat: %6\nAccess: %7")
717  .arg(qRound(m_pluginWidget->storageInfo()->bytesTotal() / qPow(1024, 3)))
718  .arg(qRound(m_pluginWidget->storageInfo()->bytesAvailable() / qPow(1024, 3)))
719  .arg(QString(m_pluginWidget->storageInfo()->device()))
720  .arg(m_pluginWidget->storageInfo()->displayName())
721  .arg(m_pluginWidget->storageInfo()->name())

734 ## 支持右键菜单
735 
736 增加右键菜单功能需要实现以下两个接口：
737 
738 - itemContextMenu
739 - invokedMenuItem
740 
741 ``` c++
742 // homemonitorplugin.h
743 public:
744  const QString itemContextMenu(const QString &itemKey) override;
745  void invokedMenuItem(const QString &itemKey, const QString &menuId, const bool checked) override;
746 ```
747 
748 ``` c++
749 // homemonitorplugin.cpp
750 const QString HomeMonitorPlugin::itemContextMenu(const QString &itemKey)
751 {
752  Q_UNUSED(itemKey);
753 
754  QList<QVariant> items;
755  items.reserve(2);
756 
757  QMap<QString, QVariant> refresh;
758  refresh["itemId"] = "refresh";
759  refresh["itemText"] = "Refresh";
760  refresh["isActive"] = true;
761  items.push_back(refresh);
762 
763  QMap<QString, QVariant> open;
764  open["itemId"] = "open";
765  open["itemText"] = "Open Gparted";
766  open["isActive"] = true;
767  items.push_back(open);
768 
769  QMap<QString, QVariant> menu;
770  menu["items"] = items;
771  menu["checkableMenu"] = false;
772  menu["singleCheck"] = false;
773 
774  // 返回 JSON 格式的菜单数据
775  return QJsonDocument::fromVariant(menu).toJson();
776 }
777 
778 void HomeMonitorPlugin::invokedMenuItem(const QString &itemKey, const QString &menuId, const bool checked)
779 {
780  Q_UNUSED(itemKey);
781 
782  // 根据上面接口设置的 id 执行不同的操作
783  if (menuId == "refresh") {
784  m_pluginWidget->storageInfo()->refresh();
785  } else if ("open") {
786  QProcess::startDetached("gparted");
787  }
788 }
789 ```
